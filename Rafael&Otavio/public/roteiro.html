1 Visão Geral do Ambiente

O ambiente é composto por 4 contêineres Docker orquestrados via Docker Compose, conforme exigido no trabalho. Cada contêiner tem uma responsabilidade específica e eles se comunicam através de uma rede interna criada automaticamente pelo Compose.

Contêiner Imagem Função Acesso externo
c2-app build local Servidor web Node.js + Frontend localhost:8080
c2-postgres postgres:16-alpine Banco de dados relacional Nenhum — só interno
c2-adminer adminer:latest Administração gráfica do banco localhost:8081
c2-portainer portainer-ce:latest Gerenciamento gráfico dos contêineres localhost:9000
Requisito atendido — banco de dados isolado O contêiner c2-postgres não possui mapeamento de porta (ports:), portanto é acessível apenas pelos outros contêineres dentro da rede app-network, nunca por máquinas externas.

2 Explicação do docker-compose.yml

O arquivo docker-compose.yml é o coração do ambiente. Abaixo cada bloco é explicado:

2.1 Serviço postgres — Banco de Dados

postgres:
  image: postgres:16-alpine          # imagem oficial PostgreSQL, Alpine = mais leve
  container_name: c2-postgres
  restart: unless-stopped             # reinicia automaticamente se cair
  environment:
    POSTGRES_USER: c2user
    POSTGRES_PASSWORD: c2password
    POSTGRES_DB: c2db
  volumes:
    - postgres_data:/var/lib/postgresql/data   # ← persistência dos dados
  networks:
    - app-network                          # ← sem "ports:", banco inacessível externamente
  healthcheck:
    test: ["CMD-SHELL", "pg_isready -U c2user -d c2db"]
    interval: 5s
    retries: 10                           # app só sobe quando banco está saudável
Por que volume nomeado? O volume postgres_data garante que os dados não se percam ao desligar ou reiniciar o contêiner — atendendo o requisito de armazenamento definitivo.

2.2 Serviço app — Servidor Web

app:
  build: .                               # constrói imagem a partir do Dockerfile local
  ports:
    - "8080:3000"                        # porta 8080 da máquina → porta 3000 do contêiner
  depends_on:
    postgres:
      condition: service_healthy        # aguarda o banco estar pronto
  environment:
    DATABASE_URL: postgresql://c2user:c2password@postgres:5432/c2db
    JWT_SECRET:   docker_jwt_secret_2024
    NODE_ENV:     production
  networks:
    - app-network

2.3 Serviço adminer — Administração do Banco

adminer:
  image: adminer:latest               # interface web para gerenciar o banco
  ports:
    - "8081:8080"                       # acessível em localhost:8081
  depends_on:
    - postgres
  networks:
    - app-network                       # na mesma rede → acessa o banco pelo nome do container

2.4 Serviço portainer — Gerenciamento dos Contêineres

portainer:
  image: portainer/portainer-ce:latest
  ports:
    - "9000:9000"                       # dashboard em localhost:9000
  volumes:
    - /var/run/docker.sock:/var/run/docker.sock  # acesso ao daemon Docker
    - portainer_data:/data             # persiste configurações do Portainer

2.5 Rede e Volumes

volumes:
  postgres_data:   # dados do PostgreSQL — persiste ao desligar o contêiner
  portainer_data:  # configurações do Portainer

networks:
  app-network:
    driver: bridge  # rede bridge isolada; postgres não expõe porta → inacessível externamente

3 Como Funciona o Dockerfile

O Dockerfile descreve como a imagem da aplicação Node.js é construída:

  1. Instala dependências — npm install (camada cacheada para builds rápidos)
  2. Gera o Prisma Client — npx prisma generate (cliente para PostgreSQL)
  3. Compila TypeScript — npm run build → pasta dist/
  4. Na inicialização — npx prisma db push cria as tabelas no banco, depois node dist/server.js
Stack da aplicação web Node.js 20 + TypeScript + Express.js + Prisma ORM (PostgreSQL) + JWT + Zod. O frontend (SPA HTML/CSS/JS + Bootstrap 5) é servido pelo próprio Express como arquivo estático em localhost:8080.

5 Comandos Úteis

ComandoO que faz
docker compose up --buildConstrói as imagens e sobe todos os contêineres
docker compose up -dSobe em background (detached)
docker compose downPara e remove os contêineres (volumes são mantidos)
docker compose down -vPara e remove contêineres e volumes (apaga dados)
docker compose logs -f appAcompanha os logs da aplicação em tempo real
docker compose psLista contêineres e seus status
